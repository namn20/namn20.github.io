ISMS-P 인증 통과의 가장 거대한 산맥이자 기술적 통제 사항이 집중적으로 몰려 있는 영역이 바로 **2.10 (시스템 및 서비스 보안관리)**입니다. 이 영역은 방화벽과 같은 전통적인 보안 솔루션부터 클라우드 특화 보안, 업무용 PC 보안, 취약점 패치 및 악성코드 예방에 이르는 정보시스템 보안의 A부터 Z까지 모든 엔지니어링 통제를 포괄합니다.

특히 멀티 클라우드를 도입한 중견기업에게는 CSP별 설정 취약점 진단 및 컨테이너/서버리스 보안을 다루는 '클라우드 보안(2.10.2)' 항목이 컴플라이언스 획득의 성패를 가르는 핵심 분수령이 됩니다.

이번 글에서는 2.10의 9개 세부 항목을 멀티 클라우드 및 최신 인프라 트렌드에 맞춰 알아보고, 실전 시험 문제를 복원한 시나리오 퀴즈를 정리해 보겠습니다.

---

## 1. 2.10.1 보안시스템 운영 & 2.10.2 클라우드 보안

### 📌 인증기준 개요
- **2.10.1 보안시스템 운영**: 침입차단시스템(방화벽), 침입탐지시스템(IDS/IPS), 웹방화벽(WAF) 등 보안시스템을 도입·운영하고, 룰셋의 정기적 검토와 접근 제한 대책을 세워야 합니다.
- **2.10.2 클라우드 보안**: 클라우드 서비스를 이용하는 경우, 클라우드 공동 책임 모델에 따른 보안 범위를 규정하고, **클라우드 서비스 설정 취약점과 보안 자격 증명(Access Key 등)을 안전하게 관리**해야 합니다.

### ☁️ 멀티 클라우드 환경에서의 핵심 고려사항
- **클라우드 특화 방화벽 및 WAF**: AWS WAF, Cloudflare WAF 등의 정책을 정기적으로 업데이트하고, 비정상 트래픽 탐지 시 즉각 알림이 전송되도록 가동해야 합니다.
- **CSPM (Cloud Security Posture Management) 도입**: 멀티 클라우드 전반의 보안 설정 오류(예: S3 Public 개방, 취약한 암호화 설정 등)를 실시간 감사하고 대시보드화하여 통제해야 합니다.
- **API 자격증명 노출 통제**: 임직원이 AWS Access Key 등 크레덴셜을 텍스트 형태로 로컬에 저장해 활용하는 행위를 엄격히 금지하고, 대신 IAM Role이나 임시 자격 증명(STS) 방식을 적용해야 합니다.

### 📝 실전 시험 대비 (기출 복원 시나리오)
**[상황] 심사원이 A기업의 멀티 클라우드 보안 아키텍처 및 설정을 확인하고 있다.**
* **심사원:** "귀사가 사용 중인 클라우드 계정들의 보안 설정 취약점을 점검하는 체계가 갖춰져 있나요?"
* **담당자:** "인프라팀에서 매주 수동으로 주요 방화벽 설정을 검토하고 있어 문제가 없습니다."
* **심사원:** "최근 개발자가 개인 퍼블릭 GitHub 저장소에 소스 코드를 올리면서 실수로 AWS의 루트 계정 액세스 키(Access Key)를 하드코딩하여 유출하는 사고가 타사에서 빈번했습니다. 이 키 유출 사고를 예방하거나 즉각 감지하여 회수하는 보안 통제가 작동하고 있습니까?"
* **담당자:** "개발자들에게 하드코딩 금지 메일을 배포하긴 했으나, 시스템적으로 소스 저장소(GitHub 등) 상의 유출 여부를 감시하거나 API 키 유출을 실시간 탐지하는 기술적 수단은 도입되어 있지 않습니다."

**Q. 위 상황에서 심사원이 식별할 수 있는 결함으로 가장 올바른 것은?**
1. 2.10.1 보안시스템 운영 - 웹 방화벽(WAF)의 트래픽 처리 성능 미달
2. **2.10.2 클라우드 보안 - 클라우드 서비스 자격 증명(Access Key)에 대한 무단 노출 방지 및 실시간 탐지·통제 대책이 수립되지 않음 (정답)**
3. 2.8.3 시험과 운영환경 분리 - 개발자가 GitHub를 사용하여 코드를 빌드함
4. 2.10.3 공개서버 보안 - 외부 소스 코드 저장소 서버에 방화벽 정책이 누락됨

**해설:** 클라우드 관리 자격 증명(Access Key)은 인프라 조작 권한을 직접 가지므로, 하드코딩 금지, 주기적인 키 갱신, GitHub Secret Scanning과의 연동을 통한 유출 시 자동 차단/취약점 스캔 등이 필수입니다. 이를 방치한 경우 **2.10.2 클라우드 보안** 결함으로 직결됩니다.

## 2. 2.10.3 공개서버 보안 & 2.10.5 정보전송 보안

### 📌 인증기준 개요
- **2.10.3 공개서버 보안**: 외부 인터넷에 서비스를 제공하는 웹 서버, 메일 서버 등(공개서버)에 대해 불필요한 서비스 포트 및 시스템 에러 메시지 노출을 차단하고 안전한 설정(Hardening)을 유지해야 합니다.
- **2.10.5 정보전송 보안**: 메일, 메신저, 협업 도구 등을 통해 개인정보나 중요 정보를 외부로 전송할 경우, 승인 절차를 수립하고 파일 암호화 조치 등을 취해야 합니다.

### ☁️ 멀티 클라우드 환경에서의 핵심 고려사항
- **에러 핸들링 및 가명화**: 외부 공개 웹 페이지에서 에러가 발생했을 때 내부 DB 연결 상태나 스택 트레이스(Stack Trace) 정보가 노출되지 않도록 표준 에러 처리 페이지를 웹서버 및 WAS 단에 세팅해야 합니다.
- **민감 정보 전송 시 승인 및 암호화**: 사내용 SaaS 협업 툴(Slack, Teams 등)을 통해 주민등록번호나 회사 기밀 데이터가 암호화되지 않은 채 파일로 전송되는 행위를 모니터링(DLP 등 연동)하고 필터링해야 합니다.

## 3. 2.10.6 업무용 단말기기 보안 & 2.10.7 보조저장매체 관리

### 📌 인증기준 개요
- **2.10.6 업무용 단말기기 보안**: 임직원이 업무용으로 사용하는 PC, 노트북, 스마트 기기 등에 대해 백신 설치, 매체 제어, 화면 잠금, 보안 패치 강제 등 기술적 보안 대책을 적용해야 합니다.
- **2.10.7 보조저장매체 관리**: USB, 외장하드 등 보조저장매체를 이용한 내부 자료 유출을 통제하기 위해 보조저장매체 사용 신청 및 승인 절차를 마련하고 암호화 기능이 있는 보안 보조저장매체를 사용해야 합니다.

### ☁️ 멀티 클라우드 환경에서의 핵심 고려사항
- **EDR 및 MDM 통합 운영**: 클라우드 인프라에 직접 접속하여 설계·개발 작업을 수행하는 개발자 PC에는 기존 백신(Vaccine)보다 강력한 위협 탐지가 가능한 EDR 솔루션을 설치하고, 원격 기기 데이터 소거 기능(MDM)을 적용해 두어야 합니다.
- **USB/외장매체 전면 통제**: 사내 업무용 단말기에서 외장 매체로의 쓰기 권한을 원천 차단하고, 부득이한 반출 필요 시 사전 결재를 통해 지급되는 **'공식 보안 USB(자동 암호화 기능 탑재)'**를 통해서만 매체 복사가 이루어지도록 차단 정책을 적용해야 합니다.

## 4. 2.10.8 패치관리 & 2.10.9 악성코드 통제

### 📌 인증기준 개요
- **2.10.8 패치관리**: 운영체제(OS), 웹서버, WAS, 데이터베이스 및 주요 소프트웨어의 최신 보안 패치를 신속히 적용하기 위한 패치 관리 절차를 수립하고 이행해야 합니다.
- **2.10.9 악성코드 통제**: 컴퓨터 바이러스, 랜섬웨어 등 악성코드 감염을 방지하기 위해 백신 프로그램을 설치하고 실시간 감시 및 주기적 검사 기능을 상시 가동해야 합니다.

### ☁️ 멀티 클라우드 환경에서의 핵심 고려사항
- **Immutable Infrastructure (불변 인프라) 패치**: 클라우드 환경에서 가동 중인 수십 대의 VM(가상머신)에 매번 수동으로 패치 명령을 수행하는 방식은 리스크가 큽니다. 빌드 파이프라인에서 컨테이너 이미지(Docker)나 VM 이미지(AMI) 자체에 최신 패치를 적용한 '골든 이미지'를 빌드하고, 이를 롤링 업데이트 방식으로 재배포(Blue-Green 배포 등)하는 현대적 패치 프로세스를 구축해야 합니다.
- **주기적 패치 미이행 감사**: 미해결 보안 취약점(CVE)이 서버나 이미지 내부에서 6개월 이상 방치되는 일이 없도록 보안 취약점 스캐너(Trivy, Inspector 등)를 정기 가동해야 합니다.

### 📝 실전 시험 대비 (기출 복원 시나리오)
**[상황] 심사원이 B기업의 패치 관리 현황을 점검하고 있다.**
* **심사원:** "서비스에 사용 중인 외부 노출 리눅스 가상 서버(EC2)들의 커널 및 OS 보안 패치 현황을 검토하고 싶습니다."
* **담당자:** "저희는 주기적으로 서버 성능 점검 시 OS 커널 패치 작업을 직접 진행하고 있습니다."
* **심사원:** "실제 서버를 확인해 보니, 9개월 전에 보안 커뮤니티에 보고되어 긴급(Critical) 등급으로 분류된 Linux Kernel 원격 코드 실행 취약점(CVE-XXXX)에 대한 패치가 아직 설치되지 않은 상태로 실 서비스가 가동 중이군요."
* **담당자:** "아, 그 패치를 적용하면 서버가 재부팅되어야 하고 서비스 중단 우려가 있어 차일피일 미루다 보니 적용 시기를 놓쳤습니다."

**Q. 위 인터뷰 상황에서 도출되는 명백한 컴플라이언스 결함 사유는?**
1. 2.9.2 성능 및 장애관리 - 서비스 중단 위험에 대한 예외 승인을 득하지 않음
2. 2.10.9 악성코드 통제 - 리눅스 백신 라이선스가 만료됨
3. **2.10.8 패치관리 - 중대 및 긴급 보안 취약점에 대한 패치를 제공받았음에도 적절한 테스트 및 패치 이행 절차를 거치지 않고 장기간 방치함 (정답)**
4. 2.8.6 운영환경 이관 - 커널 이관 시 CI/CD 도구를 통과시키지 않음

**해설:** 긴급 보안 패치 통보는 제조사나 한국인터넷진흥원(KISA) 등에서 지속적으로 발송됩니다. 영향도 평가를 위해 패치가 미루어질 경우, 대체 통제안(WAF 차단 룰 적용 등)을 마련하거나 조속히 정기 작업 일정을 잡아 적용해야 합니다. 이를 아무런 대책 없이 장기간 미룬 상태로 두는 것은 **2.10.8 패치관리** 결함의 대표적 케이스입니다.

## 🚀 AI를 활용한 시스템 및 서비스 보안 자동화 의견

대형 클라우드 및 스마트 오피스 환경에서 엔드포인트와 인프라의 복잡성을 통제하려면 AI 기반의 자동 탐지 체계 도입이 강력히 요구됩니다.

### 1. AI 기반의 동적 EDR 및 XDR 솔루션 연동
- **알려지지 않은 위협(Zero-Day) 차단**: AI 패턴 모델이 단말에서 파일 실행 시 발생하는 동작 흐름을 모니터링합니다. 기존에 알려진 시그니처 파일 매칭 외에도, 갑자기 수천 개의 파일을 순차적으로 암호화하려는 행위(랜섬웨어 징후)나 윈도우 내부 프로세스를 우회 제어하려는 움직임을 감지하여 프로세스를 즉시 강제 킬(Kill)하고 격리시킵니다. (2.10.9 대응)

### 2. 패치 영향도 예측 및 스마트 롤백 패치 시스템
- **패치 안정성 예측 모델**: 보안 패치 번호를 입력하면 대량의 커뮤니티 데이터와 오픈소스 롤백 이력을 AI 분석하여 **"이 패치는 현재 가동 중인 Java 17 및 Nginx 1.20 버전 환경에서 2.4%의 성능 저하 또는 충돌 리스크를 유발합니다"**라는 경고와 함께 안전한 마이그레이션 가이드를 보안팀에 리포팅하여 빠르고 안정적인 최신 패치 이행률을 보장합니다. (2.10.8 대응)

시스템 및 서비스 보안관리는 ISMS-P의 심장을 구성하는 실무적 핵심 보안 영역입니다.

| 항목 | 핵심 키워드 | 클라우드 환경 핵심 체크리스트 |
| :--- | :--- | :--- |
| **2.10.1** 보안시스템 운영 | 방화벽, IPS, WAF | 클라우드 WAF 정책 및 차단 이력 정기 검토, 보안 장비 정기 점검 |
| **2.10.2** 클라우드 보안 | CSP 설정 진단, 자격증명 | CSPM을 통한 취약점 상시 점검, AWS Access Key 하드코딩 제한 |
